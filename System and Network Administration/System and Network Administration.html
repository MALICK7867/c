Domain Name System (DNS)

What is Domain Name

Domain name is a way to identify and locate computers connected to internet. Every organisation has its own domain name and no two organisation have same domain name. A domain name consist of two or more components saperated by period called dot (.). A common example of domain name is

  • www.yahoo.com
  • www.google.com
Once a domain name is established, subdomains can be created within the domain. For example: the domain name for a university may be "ue.edu.pk" and its subdomain for a saperate campus may be "fsd.ue.edu.pk"

Purpose of Naming

Addressed are used to locate objects. Here objects are defined as computers or devices connected to the internet. Instead of remembering IP address of all computers we can remember a name given to the computer which we refer as domain name of the system. It makes you to go to a specific system more easily and you do not have to remember the IP name or assign a static IP to all systems. DNS is a system we use to map from names to resources of several types.

Structure of Domain Name

A domain name has a following structure:

  • last_name subdomain second_level_domain top_level_domain

Top Level Domains

Top level domains are classified into three types. These are:

  1. Organisation or Generic Domain
  2. Geographic or Country Domain
  3. Reverse Domain

Organisation or Generic Domain

It consist of three character code which indicates the primary function of the organisation or their generic behaiviour. Most commonly used top level domains are:

  • .com for commercial sites eg www.google.com
  • .net for networking sites eg www.zedge.net
  • .gov for government sites eg www.newjersy.gov
  • .edu for educational sites eg www.ue.edu
  • .org for non-commercial organisations eg www.eklavya.org
  • .mil for military organisations eg www.dod.mil
  • .int for international organisations eg www.itu.int

Geographical or Country Domains

It consists of two characters which represents different countries all over the world. These codes are standardized by International Standard Organisation (ISO). Some examples of country domains are:

  • .in for India
  • .pk for Pakistan
  • .jp for Japan
  • .us for United States
  • .fr for France
  • .it for Italy
  • .au for Australia

Reverse Domain

Reverse domain, also known as reverse DNS (Domain Name System), is a process that maps IP addresses to domain names. While standard DNS translates human-friendly domain names into IP addresses, reverse DNS performs the opposite, providing a way to associate an IP address with a domain name. Example: Suppose you have the IP address 203.0.113.1 and want to find the corresponding domain name using reverse DNS. You might discover that the reverse DNS lookup reveals the domain name "example.com." This mapping allows network administrators to identify the hostname associated with an IP address, aiding in troubleshooting, security, and network management.

Working of DNS

Each domain name has a corresponding IP address assigned or linked with it. When a user enter the domain name in the address bar, the corresponding IP address is supplied to the user. This type of translation of domain name into an IP is done by a system called Domain Name System. Mainly Domain name system is a colection of database that contain information about the domain names and IPs linked to them.

When an application program needs to communicate with other computers, it needs to translate the name and the other computer into its IP address. The application progarm that request the service becomes the client of DNS. It then sends the request to DNS server. The server looks up the name and then returns the correct IP address to the client. A large number of DNS servers may get involved in this process of getting correct IP address. After receiving the correct IP address, computers start to communicate with eachothers.

Features of DNS

Global Distribution

The data in a DNS server is maintained locally but it is retrieved globally. There is no single DNS server that has all DNS data. Any device can look into DNS and DNS lookups can be performed by any device that sends request to the DNS. The site that is often used again again can be cacheable to improve performance.

Scalability

There is no size limit of database in a DNS server and a server may have over 20 millions names. Atleast 24,000 queries can be handled per second and there is no limit to the number of queries. Querires are distributed among Masters, Slaves and Caches.

  • Master: The master DNS server is the authoritative source for a particular DNS zone. It holds the original and definitive DNS records for a domain. This server is where DNS administrators create, update, and maintain DNS records for the domain. Changes made on the master server are propagated to secondary (slave) servers. Typically, these servers are responsible for the primary DNS zone and its records.
  • Slaves: Slave DNS servers, also known as secondary DNS servers, obtain DNS zone data from the master DNS server. They act as backups and help distribute the load for the same DNS zone. In the event that the master server becomes unavailable, the slave servers can continue to provide DNS resolution for the domain. Slave servers periodically synchronize their data with the master server to ensure they have up-to-date DNS information.
  • Cache: A DNS cache server, also referred to as a caching DNS resolver, doesn't host authoritative DNS records for any specific domain. Instead, it temporarily stores recently queried DNS information in its cache. When a user's device requests a domain resolution, the cache server first checks its cache for the answer. If the information is present, it returns the result, speeding up the resolution process. If the data is not in its cache, it queries authoritative DNS servers to obtain the answer and then stores it for future reference. Caching DNS servers help reduce the load on authoritative servers and improve overall DNS query efficiency.

Dynamicity

Database of DNS sever can be updated dynamically like add/delete/modify any record. There is a master database of all DNS servers and only this server is dynamically updated which makes a single point of faliure.

Relibility

When master server is updated, it automatically update all slave servers and data is copied to multiple slaves. A client can send query to master server and any copy at a slave server. DNS protocols can use either TCP or UDP. If it is set to UDP, DNS protocol handle retransmission, sequencing, etc.

Loose Coherence

The database in DNS is always internally consistent. Each version of a subset of database (a zone) has a serial number. The serial number change on every incriment on the database. Changes to master copy of database are replicated or incrimented according to the timing set by zone administrator. He also set the timeout for data expiration of cached data.

Key Based Authentication (SSH)

Introduction

When an Ubuntu system is first installed, it is not configured by default to allow remote command-line access via Secure Shell (SSH) connections. When installed, SSH provides password protected and encrypted access to the system for the root account and any other users added during the installation phase.

Overview of SSH

SSH is designed to allow secure remote access to systems for the purposes of gaining shell access and transferring files and data. SSH can also be used to provide a secure tunnel through which remote access to the GNOME desktop can be achieved over a network connection. A basic SSH configuration consists of a client and a server.

A user might, for example, use an SSH client running on a Linux, Windows or macOS system to connect to the SSH server running on an Ubuntu system to gain access to a shell command-line prompt or to perform file transfers. All of the communications between client and server, including the password entered to gain access, are encrypted to prevent outside parties from intercepting the data.

The inherent weakness in a basic SSH implementation is that it depends entirely on the strength of the passwords assigned to the accounts on the system. If a malicious party is able to identify the password for an account the system becomes vulnerable. This weakness can be addressed by implementing SSH key-based authentication.

Encryption In SSH

SSH key-based authentication makes use of asymmetric key encryption to add an extra layer of security to remote system access. This is completly based on the concept of using a pair of keys, one private and one public. In a public key encryption system, the public key is used to encrypt data that can only be decrypted by the owner of the private key.

In the case of SSH key-based authentication, the private key is held by the host on which the SSH client is located while the corresponding public key resides on the system on which the SSH server is running. It is important to protect the private key, since ownership of the key will allow anyone to log into the remote system.

Installing SSH

If the SSH server is not already installed and running on the system, it can be added using the following commands:

Installation of SSH server:

# apt install openssh-server

Starting SSH server's Services:

# systemctl start sshd.service

Enabling SSH server's Servers:

# systemctl enable sshd.service

Setting Remote Access Linux

The first step in setting up SSH key-based authentication is to generate the key pairs on the client system. If the client system is running Linux or macOS, this is achieved using the ssh-keygen utility:

# ssh-keygen
Generating public/private rsa key pair.
Enter file in which to save the key
(/home/username/.ssh/id_rsa):

Press the Enter key to accept the default location for the key files. This will place two files in the .ssh sub-directory of the current user's home directory. The private key will be stored in a file named id_rsa while the public key will reside in the file named id_rsa.pub.

Next, ssh-keygen will prompt for a passphrase with which to protect the private key. For better security, use of a passphrase is recommended. Then finally, the ssh-keygen tool will generate the following output indicating that the keys have been generated:

Enter passphrase (empty for no passphrase):
Your identification has been saved in /home/neil/.ssh/id_rsa.
Your public key has been saved in /home/neil/.ssh/id_rsa.pub.
The key fingerprint is:
SHA256:FOLGWEEGFIj WnCT5wtTOv5VK4hdimzWghZiz UEMYbfo
(username)@(hostname)
The key's randomart image is:
+---[RSA 2048]----+
|.BB+=+*.. |
|o+B= * . . |
|===.. + . |
|*+ * . . |
|.++ o S |
|..E+ * o |
| o B * |
| + + |
| . |

The next step is to install the public key onto the remote server system. This can be achieved using the ssh-copy-id utility as follows:

$ ssh-copy-id username@remote_hostname

For Example:

$ ssh-copy-id neil@192.168.1.100
/usr/bin/ssh-copy-id: INFO: Source of key(s) to be installed:
"/home/neil/.ssh/id_rsa.pub"
/usr/bin/ssh-copy-id: INFO: attempting to log in with the new
key(s), to filter out any that are already installed
/usr/bin/ssh-copy-id: INFO: 1 key(s) remain to be installed --
if you are prompted now it is to install the new keys
neil@192.168.1.100's password:
Number of key(s) added: 1
Now try logging into the machine, with: "ssh
'neil@192.168.1.100'"
and check to make sure that only the key(s) you wanted were
added.

Once the key is installed, test that the authentication works by attempting a remote login using the ssh client:

$ ssh -l (username) (hostname)

If the private key is encrypted and protected with a passphrase, enter the phrase when prompted to complete the authentication and establish remote access to the Ubuntu system:

Enter passphrase for key '/home/neil/.ssh/id_rsa':
Last login: Thu Feb 21 13:41:23 2019 from 192.168.1.101
[neil@demosystem02 ~]$

Once you have verified that key-based authentication works, log into the remote system, edit the /etc/ssh/ssh_config file and change the PasswordAuthentication setting to no:

PasswordAuthentication no

To reload the service of SSH, type the following command:

# systemctl restart sshd.service

From this point on, it will only be possible to remotely access the system using SSH key-based authentication.

Authentication Using GUI Interface PuTTY

PuTTY tool is widely used for remote access to any server if the operating system do not support OPEN-SSH. The first step in using PuTTY is to download and install it on any Windows systems that need an SSH client. PuTTY is a free utility and can be downloaded using the following link: www.chiark.greenend.org.uk/

Download the Windows installer executable that matches your Windows system then execute the installer to complete installation.

If a private key already exists on another system, create the .ssh folder in the home folder of the current user and copy the private id_rsa key into it.

Next, the private key file needs to be converted to a PuTTY private key format file using the PuTTYgen tool. Locate this utility in the Windows Start menu and launch it:

Click on the load button to load any existing key which was created previously. Once located, select the file and load it into PuttyGen and when required, enter the paraphrase/passphrase. Once the private key has been imported, save it as a PuTTY key file by clicking on the Save Private Key button.

Launch PuTTY from the Start menu and enter the IP address or host name of the remote server into the main screen before selecting the Connection -> SSH -> Auth category in the left-hand panel as highlighted:

Click on the Browse button next to the Private key for authentication field and navigate to and select the previously saved PuTTY private key file.

Private key in PuTTY

If keys do not already exist, they can be created using the PuTTYgen tool which is included with the main PuTTY installation. To create new keys, launch PuttyGen and click on the Generate button highlighted:

Once the keys have been generated, save the files to suitable locations using the Save public key and Save private key buttons. You can also change the size of the key to any size whichby default will be 2048. You can also select the type of algorithm you want to use in the encryption process of keys.
