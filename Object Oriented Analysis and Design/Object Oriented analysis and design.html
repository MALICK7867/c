Requirement Analysis Process

Requirement analysis is a critical phase in the SDLC. It involves gathering, documenting, and analyzing the needs and expectations of stakeholders to define the features and functionalities of a software system or application. The primary goal of requirement analysis is to ensure that the final software product meets the intended objectives and satisfies user needs.

The process of studying and analyzing the customer and the user/stakeholder needs to arrive at a definition of software requirements. Provides software designer with a model of:

  • System Information
  • System Function
  • System Behavior
Model can be translated to data, architectural, and component-level designs and expect to do a little bit of design during analysis and a little bit of analysis during design.

Requirement Analysis Objectives

  1. Identify customer's needs
    • The fundamental objective is to gain a deep understanding of what various stakeholders, including end-users, customers, and business owners, need and expect from the software. This involves identifying their goals, challenges, and desired functionalities.
  2. Evaluate system for feasibility
    • Evaluating a system for feasibility is an essential step in the early stages of a software development project. It is typically done during the project initiation phase, before significant resources are committed. The purpose of feasibility analysis is to determine whether the proposed software system is worth pursuing, given various constraints and factors.
  3. Perform economic and technical analysis
    • It is an essential part of evaluating the feasibility of a software development project. These analyses help assess whether the project is financially viable and technically feasible.
  4. Allocate functions to system elements
    • It is a critical step in system design and architecture. This process involves assigning specific functions or responsibilities to different components or modules within a system to ensure that the system as a whole can achieve its intended goals and meet its requirements.
  5. Establish schedule and constraints
    • It is a critical part of project management and planning. A well-defined schedule helps ensure that the project stays on track and is completed within the specified time frame, while constraints set boundaries and limitations that the project must operate within.
  6. Create system definitions
    • It is an important step in the early stages of a project, as it helps ensure that all stakeholders have a common understanding of what the system is supposed to achieve and how it will function. System definitions provide a clear and concise description of the project scope and objectives.

What is Requirement Validation

Requirement validation is a crucial step in the software development process. It is the process of reviewing and evaluating the gathered requirements to ensure that they are accurate, complete, consistent, and meet the needs of the stakeholders. The primary goal of requirement validation is to identify and rectify any issues or discrepancies in the requirements before they are used as the basis for designing and building the software system.
Requirements error costs are high so validation is very important. Fixing a requirements error after delivery may cost up to 100 times the cost of fixing an implementation error.

Software Requirement Specification (SRS)

SRS is a complete description of the behavior of the system to be developed. It is basically an organization's understanding (in writing) of a customer or potential client's system requirements and dependencies at a particular point in time prior to any actual design or development work. It states in percise and explicit language those functions and capabilities a software system must provide, as well as states any required constraints by which the system must be abide. SRS is a blueprint for completing a project with as little cost growth as possible.

Characteristics of SRS

Some Characteristics of SRS are:

  • Complete: SRS defines precisely all the go-life situations that will be encountered and the system's capability to address them.
  • Consistent: SRS capability functions n performance level are compatible, and the features don't negate those capability functions.
  • Accurate: SRS precisely defines the system's capability in a real-world environment.
  • Modifiable: The logical, hirarchical structure of SRS should facilitate any necessary modifications.
  • Ranked: Individual req of SRS r hierarchically arranged according to stability, security, perceived ease/difficulty of implementation, or other parameters that help in design.
  • Testable: SRS must be stated in such a manner that unambiguous assessment criteria.
  • Traceable: Each req in SRS must be uniquely identified to a source (use case, gov reg, industry standard, etc).
  • Unambiguous: SRS must contain req statement that can be interpreted in one way only.
  • Valid: All parties n project participants can understand, analyze, accept, or approve the SRS.
  • Verifiable: SRS is consistent from 1 level of abstraction to another.

Use Case Modeling

Use case modeling is a technique in software engineering and systems engineering that helps define, clarify, and document the interactions between various actors (such as users or other systems) and a system under development. It is a fundamental part of the Unified Modeling Language (UML), which is a standardized modeling language used in software development. Use case modeling focuses on capturing the functional requirements of a system from a user's perspective.

Difference Between Goals and Objective

Goals are typically broader, more overarching, and focused on the long term. They represent the ultimate outcome you aim to achieve. Goals are often less specific and more general in nature. They provide a direction but may not detail how to achieve the desired outcome.
Objectives are more specific, concrete, and focused on the short to medium term. They outline precisely what needs to be achieved to reach a goal. Objectives are actionable and often require specific tasks or actions to be taken to achieve them.

  • Goal - Long term
  • Objective - Short term

Advantages of use-case modeling

Some advantages of use case modeling are discussed as:

  • Requirements Clarification: It helps stakeholders and developers clearly understand how the system will be used and what functionality it should provide.
  • Communication: Use case diagrams provide an effective way to communicate system behavior and requirements to all project stakeholders.
  • Testing and Validation: Use cases can serve as the basis for test cases, ensuring that the system meets its intended functionality.
  • Project Planning: Use case diagrams can aid in project planning and task allocation by breaking down system functionality into manageable components.
  • Use Case Diagram: A use case represents a specific interaction between an actor (user or system) and the system being developed. It describes a particular functionality or feature of the system from the perspective of how it will be used.
  • Actor: An actor is an external entity that interacts with the system. Actors can be users, other systems, or even hardware components. Each actor typically has specific roles or responsibilities within the system.
  • System Boundary: The system boundary defines the scope of the system being modeled. It separates the system from its external actors. Use cases take place within this boundary.
  • Relationships: Use cases can have relationships with one another. Common relationships include "extends" (for optional behavior) and "includes" (for common behavior shared between use cases). These relationships help to modularize and organize the use case model.
  • Generalization: Use cases can be organized hierarchically using generalization relationships. This is similar to inheritance in object-oriented programming, where one use case (the child) inherits behavior from another (the parent).
  • Diagram: Use case diagrams are graphical representations of use cases, actors, and their relationships. These diagrams provide a visual overview of how the system interacts with its users or external systems.
